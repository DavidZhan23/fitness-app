# 移动端社区评论回复防回归清单

适用范围：`/community/:userId` 页面内「当日评论」模块（`DayCommentSection`）。

触控端（iOS / Android / 华为）统一 **inline compose**；桌面端保留目标评论 `scrollIntoView` + `focus({ preventScroll: true })`。

## 一、测试环境

| 平台 | 设备 / 入口 |
|------|-------------|
| **iOS** | 至少 1 台 iPhone；PWA（主屏幕）+ Safari |
| **Android / 华为** | 华为真机优先；华为 Chrome、华为浏览器、Chrome |
| **桌面** | Chrome（鼠标，非触控 coarse） |
| 网络 | 正常网络 + 慢网（4G 弱网） |

## 二、发布前必测主流程（P0）

### iOS / Android / 华为（触控）

1. 进入社区 → 打开任意成员详情页（带评论）
2. 在评论列表中点击「回复」
3. 期望：输入框可见、可聚焦、键盘正常弹出
4. 输入文字并发送
5. 期望：回复成功落库，列表立刻出现新回复，输入框状态恢复正常
6. **收起键盘后**，底部「写一句鼓励…」输入条仍可见

### 华为 Chrome / 华为浏览器（P0，阻断发布）

与用户反馈场景一致（社区详情页、评论区、已有多条回复）：

1. 滚到评论区，点某条评论「回复」
2. 输入框可见（placeholder 为 `回复 @昵称…`）
3. 键盘弹出后仍可输入（输入条不被键盘完全挡住）
4. 发送成功，新回复出现在列表
5. 收起键盘后，底部 compose 条仍在，可再次点「回复」或发新评论

判定标准：以上任一步失败，即阻断发布。

### 桌面 Chrome（P1）

1. 点「回复」→ 目标评论滚入视口 → 输入框聚焦
2. 输入并发送成功

## 三、关键场景回归（P1）

### 1) 键盘弹出/收起

- 点击「回复」后弹出键盘，输入框不应消失
- 手动收起键盘后，输入框仍在当前页面可见
- 再次点击「回复」，输入框可再次聚焦

### 2) 页面在中部滚动时回复

- 将页面滚到中段（不是顶端/底端）
- 点击中段评论的「回复」
- 期望：页面滚动与输入框定位稳定，不闪烁、不跳飞；compose 滚入可视区

### 3) 连续切换回复对象

- 连点不同评论的「回复」（A → B → C）
- 期望：placeholder 正确切换为 `回复 @昵称`
- 输入框不中断、不消失、不丢焦

### 4) 发送中状态

- 点击发送后（网络慢时）按钮进入发送态
- 期望：不会重复提交；发送成功后输入清空

### 5) 长评论列表

- 在 20+ 条评论的页面执行回复
- 期望：输入框和列表表现稳定，无明显卡顿

### 6) 旋转与返回

- 横竖屏切换后再点「回复」（移动端）
- 从详情页返回社区再进入，重复测试回复
- 期望：回复行为一致，无「首次可用、二次失效」

## 四、兼容检查（P2）

- iOS：Safari 地址栏展开/收起状态下，回复输入正常
- iOS：PWA 与 Safari 表现一致
- 深色/浅色主题下，输入框与按钮可见性正常
