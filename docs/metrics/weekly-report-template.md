# 每周质量报告模板

> 本文件描述周报的结构规范、AI 解读表达要求，供 `weeklyReport.js#buildReportMd` 参考，
> 也供人工补充报告时遵循。
>
> 实际报告存放于 [`docs/reports/weekly/`](../reports/weekly/)，命名格式 `YYYY-Www.md`。

## 报告结构（8 段）

---

### 1. 本周结论

一句话总结本周最重要的交互质量状况。

**格式**：`> 路由切换 P95 **XXX ms**（良好/中等/偏高）；AI 估算成功率 **XX%**（良好/中等/偏低）。`

---

### 2. 核心指标

用表格呈现三大类指标（路由切换 / 页面加载 / AI 估算），含环比变化。

```
### 路由切换
| 指标 | 本周 | 环比 |
|------|------|------|
| 次数 |  | — |
| P50  |  | — |
| P95  |  | +/-X% |

### 页面加载
| 指标 | 本周 | 环比 |
| …    |      |      |

### AI 估算
| 指标         | 本周 | 环比 |
|--------------|------|------|
| 总量         |      | — |
| 成功         |      | — |
| 超时         |      | — |
| 错误         |      | — |
| 成功率       |      | +/-X% |
| 超时率       |      | — |
| Fallback 完成率 |   | — |
```

样本量 < 5 时标 `N/A（数据不足）`，不填具体数字。

---

### 3. 异常路由（P95 Top 5）

列出本周 P95 最慢的路由切换路径（`route_from → route_to`）。

```
| 路径 | 次数 | P95 |
|------|------|-----|
| `/` → `/community` | 12 | 1820 ms |
```

样本不足时写：`> 本周样本不足，暂无异常路由数据。`

---

### 4. 页面加载分析

列出加载 P95 最慢的页面，与上周对比。

---

### 5. AI 估算分析

- 本周共发起 AI 估算 **X** 次：成功 X、超时 X、错误 X。
- AI 未成功后用户仍完成打卡（fallback complete）**X** / X 次（X%）。

---

### 6. 可能原因

> **AI 解读表达规范（强制）**
>
> ✅ 推荐写法：
> "观察到社区页切换 P95 偏高（1820 ms），初步怀疑与全量数据重新请求和全屏 loading 有关，
> 需要结合 Network 面板或后端接口耗时进一步验证。"
>
> ❌ 不要写：
> "因为接口慢，所以页面卡。"
>
> **关键词**：初步怀疑 / 需要进一步验证 / 不排除 / 建议结合 X 验证

如果数据不足（N/A），指出需要收集哪些信息才能得出结论。

---

### 7. 优化建议

列出 3 条可落地建议，优先级从高到低，每条包含验证方法或代码改动方向：

1. **（高优先）** 具体建议，包含验证步骤
2. **（中优先）** ...
3. **（低优先）** ...

---

### 8. 下周验证目标

用可量化的 checklist 写明下周要验证的改进目标：

- [ ] route_change P95 下降至 ≤ XXX ms
- [ ] AI 估算成功率提升至 ≥ XX%
- [ ] AI fallback 完成率 ≥ 70%

---

## AI 解读 Prompt 规范

`weeklyReport.js` 中向 DeepSeek 发送的 prompt 包含以下硬性要求：

```
要求：
1. 使用严谨的可证伪表达，如"初步怀疑……需要结合 Network 面板或后端接口耗时进一步验证"
2. 禁止使用单一强因果断言，如"因为接口慢，所以页面卡"
3. 如果数据不足（标注 N/A），指出不足并说明需要收集哪些信息
```

人工补充时应遵循同样规范。

## 命名约定

- 文件名：`YYYY-Www.md`，如 `2026-W22.md`（ISO 8601 周，周一起始）
- 存放路径：`docs/reports/weekly/`
- 每份文件由 `scripts/pull-weekly-report.mjs` 从 API 拉取生成，或人工补充

## 与看板的关系

当前阶段（M4）只有 markdown 文件，不做前端看板。M5 看板可直接读：
- `weekly_reports.metrics_json`（聚合指标 JSON）
- `telemetry_events`（原始事件，按 `session_id` / `app_version` 下钻）
